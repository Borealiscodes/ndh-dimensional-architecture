# Pin Registry Protocol™
**UMA Architecture — Governance Rules for Pin Operations**  
**Version:** 1.0  
**Date:** 2026‑07‑02  
**Status:** Canonical Baseline  
**Author:** Borealis S. Hedling

## Purpose
The Pin Registry Protocol™ defines how pins are created, validated, indexed, superseded, and retired within UMA.  
It ensures that all pin operations remain constitutional, traceable, non‑drifting, and aligned with UMA’s stability architecture.

---

## 1. Scope
This protocol applies to:

- creation of new pins  
- updates to existing pins  
- supersession of pins by newer versions  
- retirement of obsolete pins  
- validation and containment of pin operations  
- registry indexing and maintenance  

It governs the **Pin Registry** and all files under `uma-core/Pins/`.

---

## 2. Pin Creation Rules

### 2.1 Preconditions
A new pin may only be created when:

- a structural workflow, anchor, or organ needs canonical reference  
- diagnostics have classified the change (constitutional, mathematical, geometric, epistemic, provenance, cross‑layer)  
- the change passes the Containment Confirmation Pass™ or triggers Pause Layer Correction Protocol™ and is resolved

### 2.2 Required Pin Fields
Every pin must include:

- name (e.g., Constitutional Workflow Pin™)  
- file path (e.g., `uma-core/Pins/Constitutional_Workflow_Pin.md`)  
- version  
- date  
- author  
- anchor description (what it pins)  
- purpose  
- status (Active, Required, Non‑Bypassable, Superseded, Retired)

### 2.3 Workflow Requirement
No pin may be created without an associated workflow description, either:

- in the pin file itself, or  
- in a referenced protocol or companion document

---

## 3. Pin Validation Rules

### 3.1 Diagnostic Validation
Before a pin is added to the registry:

- run constitutional, mathematical, geometric, epistemic, and provenance diagnostics  
- classify the pin event  
- ensure no contradictions or drift are introduced

### 3.2 Containment Validation
Apply the Containment Confirmation Pass™ to the pin event:

- Core alignment  
- constitutional integrity  
- mathematical / geometric / epistemic / provenance containment  
- cross‑layer containment  
- pause requirement

If containment fails → activate Pause Layer Correction Protocol™ and resolve before proceeding.

---

## 4. Registry Update Rules

### 4.1 Entry Requirements
When a pin is validated, the Pin Registry must be updated with:

- file path  
- version  
- date added  
- anchor description  
- purpose  
- status  

### 4.2 Indexing
Pins must be indexed by:

- type (constitutional, architectural, workflow, diagnostic, runtime, etc.)  
- layer (constitutional, engineering, runtime, epistemic, provenance)  
- status (Active, Required, Non‑Bypassable, Superseded, Retired)

---

## 5. Supersession & Retirement Rules

### 5.1 Supersession
A pin may be superseded when:

- a newer version of the same anchor is created  
- the workflow or governance rules evolve

Supersession requires:

- full validation and containment  
- explicit registry update marking the old pin as **Superseded**  
- Roots lineage entry describing the change

### 5.2 Retirement
A pin may be retired only when:

- its anchor is no longer structurally used  
- its function is fully replaced by other canonical artifacts

Retirement requires:

- validation and containment  
- registry status set to **Retired**  
- Roots lineage entry explaining the retirement

- recorded in the Pin Registry  
- reflected in the Roots as a lineage entry  
- aligned with provenance logs where applicable
