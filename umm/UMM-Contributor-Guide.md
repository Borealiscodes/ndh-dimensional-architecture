# 🌐 **UMM CONTRIBUTOR GUIDE**  
### *How to Contribute Safely, Consistently, and Constitutionally to the UMM Repo*

```
# UMM Contributor Guide
### Practical guidelines for adding, modifying, and maintaining UMM documents.

This guide explains naming conventions, folder placement, document structure,
and contribution rules for the UMM meta‑model.
```

---

# 🧭 **1. Core Principles for Contributors**

Every contribution must follow five constitutional principles:

- **Clarity** — documents must be readable, structured, and navigable  
- **Reversibility** — nothing should trap the reader or force interpretation  
- **Non‑coercion** — no prescriptive meaning, no psychological push  
- **ND‑accessibility** — low ambiguity, low branching, gentle pacing  
- **Structural Integrity** — contributions must not break the UMM architecture  

These principles apply to *every* file in the repo.

---

# 🗂️ **2. Folder Placement Rules**

Use this rule:

> **Place the file in the lowest folder that fully describes its function.**

Examples:

- A Winter geometry file → `/umm/winter/geometry/`  
- A null‑expansion file → `/umm/null/`  
- An Atropus operational file → `/umm/atropus/`  
- A cross‑season geometry file → `/umm/geometry/`  
- A yearly structural file → `/umm/yearly/`  

If unsure, follow the **Repo Structural Index**.

---

# 🧱 **3. Naming Conventions**

All UMM files follow this pattern:

```
UMM-[SCOPE]-[SUBJECT].md
```

Where:

- **SCOPE** = Yearly, Seasonal, Winter, Null, Atropus, Geometry, etc.  
- **SUBJECT** = Spec, Map, Codex, Atlas, Schematic, Field‑Manual, etc.  

Examples:

- `UMM-WINTER-DEEP-GEOMETRY-SCHEMATIC.md`  
- `UMM-NULL-EXPANSION-ARCHITECTURE.md`  
- `UMM-ATROPUS-FIELD-MANUAL.md`  

Avoid abbreviations unless they are glyph‑level (Φ₀, Λ₀, etc.).

---

# 📄 **4. Document Structure Template**

Every UMM document should follow this structure:

1. **Title**  
2. **Subtitle**  
3. **Code‑block header** (purpose + reversibility statement)  
4. **Sections** (3–12 depending on complexity)  
5. **ASCII diagrams** (if applicable)  
6. **Guided Links** to related documents  
7. **Summary**  

This ensures consistency across the repo.

---

# 🧩 **5. When to Create a New Document**

Create a new file when:

- a concept has **structural independence**  
- a subsystem needs its own **spec, codex, map, or manual**  
- a geometry layer requires **visualization**  
- a seasonal or yearly component needs **its own mapping**  

Do **not** create a new file when:

- the content is a minor extension of an existing file  
- the concept is not structurally distinct  
- the file would duplicate existing material  

If unsure, ask:  
**Does this concept have its own constitutional identity?**

If yes → new file.  
If no → extend an existing file.

---

# 🧪 **6. Tests and Validation**

All new documents should be validated against:

- **UMM Invariants**  
- **UMM Planes**  
- **UMM Yearly Structure**  
- **Seasonal System**  
- **Null‑Expansion Architecture**  
- **Atropus Null‑Plane**  

This ensures the new file does not break the architecture.

---

# 🧊 **7. Winter / Null / Atropus Special Rules**

These three subsystems have stricter rules:

### Winter  
- no expansion  
- no texture  
- no drift  
- stillness pacing  

### Null‑Expansion  
- zero‑growth  
- zero‑accumulation  
- collapse vectors only  

### Atropus  
- anchor‑only  
- tone‑minimal  
- no branching  
- no narrative buildup  

If a file touches these systems, it must follow their invariants.

---

# 🧼 **8. Style & Tone Guidelines**

UMM documents must be:

- ND‑friendly  
- low‑ambiguity  
- low‑branching  
- reversible  
- gentle  
- symbolic, not interpretive  

Avoid:

- metaphorical overreach  
- psychological claims  
- narrative framing  
- lore‑based meaning  

UMM is a **navigation system**, not a story.

---

# 🔗 **9. Guided Link Requirements**

Every document must include:

- links to related documents  
- links to parent systems  
- links to sibling systems  
- links to deeper layers (codex, atlas, schematic)  

Examples:

- **Winter Deep Geometry Codex**  
- **Null‑Expansion Interaction Diagram**  
- **Atropus Field Manual**  

This keeps the repo navigable.

---

# 🧭 **10. Contributor Workflow**

1. Identify the subsystem  
2. Confirm invariants  
3. Choose correct folder  
4. Use naming conventions  
5. Follow document structure  
6. Add Guided Links  
7. Validate against architecture  
8. Commit with a clear message  

This workflow ensures consistency.

---

# ⭐ **11. Summary**

The UMM Contributor Guide ensures:

- structural integrity  
- navigational clarity  
- ND‑friendly pacing  
- constitutional consistency  
- predictable repo organization  

It is the **operational handbook** for anyone contributing to the UMM.
