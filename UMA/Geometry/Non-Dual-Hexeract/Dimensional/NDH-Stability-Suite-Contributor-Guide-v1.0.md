# ⭐ **1. Purpose of This Guide**  
The **NDH Stability Suite Contributor Guide v1.0** explains how contributors should:

- write new stability documents  
- update existing stability documents  
- maintain governed structure  
- preserve naming conventions  
- integrate visuals  
- cross‑link the Stability Spine  
- follow NDH terminology and tone  

This guide ensures the Stability Suite remains coherent, stable, and future‑proof.

---

# ⭐ **2. Core Contribution Principles**

### **2.1 Governed Structure**  
All stability documents must follow NDH’s governed structure:

- numbered sections  
- clear headers  
- mathematical expressions where required  
- cross‑references using Guided Links  
- no inline images  
- PNGs referenced only in dedicated visual sections  

### **2.2 Dimensional Consistency**  
All contributions must respect the eleven NDH dimensional fields (D1–D11).  
No new fields may be added without Codex‑level approval.

### **2.3 Terminology Discipline**  
All terms must match the **NDH Lexicon**.  
New terms must be added to the Lexicon before being used in any stability document.

### **2.4 Visual Governance**  
PNG visuals must:

- follow the governed naming scheme  
- be placed in the Dimensional directory  
- be referenced only in visual sections  
- be accompanied by a textual description  

Visuals must never be embedded directly in MD files.

---

# ⭐ **3. Required Document Naming Conventions**

All stability documents must follow:

```
NDH-<Component>-v<Major.Minor>.md
```

Examples:

- `NDH-Stability-Matrix-v1.0.md`  
- `NDH-Dimensional-Stability-Field-Dynamics-v1.0.md`  
- `NDH-Stability-Field-Codex-v1.0.md`  

PNG naming follows the same pattern:

```
NDH-<Component>-v<Major.Minor>.png
```

Version numbers must increment only when:

- structure changes  
- definitions change  
- visuals change  
- mathematical models change  

Cosmetic edits do **not** increment version numbers.

---

# ⭐ **4. Required Directory Structure**

All Stability Suite artifacts must reside in:

```
UMA/Geometry/Non-Dual-Hexeract/Dimensional/
```

No stability documents may be placed outside this directory.

Subdirectories are not permitted unless approved in the Codex.

---

# ⭐ **5. How to Create a New Stability Document**

Follow this exact sequence:

1. **Define the purpose**  
2. **Create the governed structure**  
3. **Add mathematical models (if applicable)**  
4. **Add dimensional references (D1–D11)**  
5. **Add drift/continuity/SID behavior**  
6. **Add cross‑references using Guided Links**  
7. **Add a visual reference section**  
8. **Add a synthesis section**  
9. **Add the document to the Stability Suite Index**  
10. **Add the document to the Changelog**  

This ensures the Stability Spine remains coherent.

---

# ⭐ **6. How to Update an Existing Stability Document**

Updates must follow this sequence:

1. **Identify the affected sections**  
2. **Preserve numbering**  
3. **Preserve terminology**  
4. **Preserve dimensional references**  
5. **Update cross‑links**  
6. **Update the Stability Suite Index**  
7. **Update the Changelog**  

If the update changes meaning or structure → increment version number.  
If the update is cosmetic → do not increment version number.

---

# ⭐ **7. How to Add a New PNG Visual**

PNG additions must follow:

1. Governed naming  
2. Placement in the Dimensional directory  
3. A dedicated visual reference section in the corresponding MD  
4. A description of what the PNG represents  
5. A link in the **Stability Visual Companion**  
6. An entry in the Changelog  

PNG files must never be embedded directly in MD documents.

---

# ⭐ **8. How to Add a New Lexicon Term**

All new terms must follow:

1. **Definition**  
2. **Etymology**  
3. **Conceptual role**  
4. **Mathematical representation (if applicable)**  
5. **Cross‑references**  
6. **Synthesis**  

Terms must be added to:

- **NDH Lexicon**  
- Stability Suite Index  
- Changelog  

Terms must be used consistently across all documents.

---

# ⭐ **9. Contributor Writing Style Requirements**

### **9.1 Tone**  
NDH stability documents must be:

- formal  
- governed  
- precise  
- dimensional  
- mathematical  
- structured  

Reference documents (Companion, Visual Companion, Quickstart, FAQ) may be:

- accessible  
- narrative  
- onboarding‑friendly  
- humorous (DimmaDome Clause applies)  

### **9.2 Prohibited Content**  
Contributors must not:

- add inline images  
- break dimensional structure  
- invent new fields  
- use undefined terminology  
- alter naming conventions  
- remove cross‑links  
- add ungoverned humor (must follow DimmaDome Clause)  

---

# ⭐ **10. Required Cross‑Linking Practices**

Every stability document must link to:

- Stability Matrix  
- Field Map  
- Field Atlas  
- Field Topology  
- Continuity Map  
- Dynamics  
- Interaction Model  
- Codex  
- Lexicon  
- Visual Companion  
- Stability Suite Index  

Cross‑links must use Guided Links.

---

# ⭐ **11. Versioning Rules**

### **Major Version Increment (v2.0, v3.0, etc.)**  
Occurs when:

- dimensional structure changes  
- mathematical models change  
- stability predicates change  
- SID modes change  
- new fields are added (rare)  

### **Minor Version Increment (v1.1, v1.2, etc.)**  
Occurs when:

- visuals are added  
- definitions are expanded  
- sections are reorganized  
- terminology is clarified  

### **No Version Increment**  
Occurs when:

- typos are fixed  
- formatting is improved  
- cross‑links are added  
- directory structure is cleaned
