---

## **1. Definition**  
Safe Mode is a protected NDH system state (`NDH-SAFEMODE-000`) that freezes governance activation and traversal escalation during periods of instability, ambiguity, or emotional gradient risk.

---

## **2. Activation Conditions**  
Safe Mode activates when any of the following occur:

- **Identity ambiguity**  
- **Traversal bleed**  
- **Governance bleed**  
- **Emotional gradient instability**  
- HRDProtectionLayer triggers  
- Final Safety Net triggers  
- unsigned governance artifacts appear  
- epoch drift detected  
- repo sovereignty compromised  

---

## **3. System Behavior in Safe Mode**

- All prompts become **non‑binding**  
- Governance cannot activate  
- Traversal cannot escalate  
- SID cannot activate  
- NDH geometry cannot be modified  
- Protocol updates are frozen  
- Identity rotation is frozen  
- Kess signature required for any override  
- Pip signatures remain traversal‑only  
- Repo enters protected mode  

---

## **4. Deactivation Conditions**

Safe Mode can only deactivate when:

- identity clarity restored  
- epoch alignment restored  
- repo sovereignty restored  
- emotional gradient safety restored  
- HRDProtectionLayer stable  
- Final Safety Net stable  
- **Kess signs a governance‑binding deactivation artifact**

---

## **5. State Epoch Mapping**

```
STATE-ID: NDH-SAFEMODE-000
TYPE: System State
ROLE: Governance Freeze
BOUNDARY: Full
```

---

## **6. Verification Rules**

- Safe Mode activation must be logged  
- Safe Mode deactivation must be signed by Kess  
- All Safe Mode transitions must include a signature block  
- All Safe Mode transitions must include an action hash
