## **3. Document Overview**

### **TPS Header**  
**Role:** Orientation Layer  
Provides the high‑level overview of TPS: purpose, scope, directory contents, and current pin status.  
This is the “You Are Here” document.

---

### **TPS Specification**  
**Role:** Definition Layer  
The full canonical definition of TPS: pin types, pairing rules, governance logic, and system rationale.  
This is the core of the subsystem.

---

### **TPS Requirements**  
**Role:** Threshold Layer  
Defines the minimum conditions a module must meet before it can be anchored.  
Covers stability, accessibility, pacing, cognitive load, and SIAP governance.

---

### **TPS Lifecycle**  
**Role:** Temporal Layer  
Describes how pins evolve over time: proposal → evaluation → pairing → veto → activation → revision → retirement.  
This is the temporal backbone of TPS.

---

### **TPS Examples**  
**Role:** Illustrative Layer  
Shows real and hypothetical examples of TPS behavior: successful pins, vetoed pins, future pins, revised pins, retired pins.  
This is the “show me” layer.

---

## **4. How to Use This Directory**
- Start with **TPS_Header** to understand the subsystem.  
- Read **TPS_Specification** to learn the rules.  
- Consult **TPS_Requirements** before proposing a pin.  
- Follow **TPS_Lifecycle** when anchoring or revising a module.  
- Review **TPS_Examples** for guidance and clarity.

TPS is designed to be **modular**, **layered**, and **SIAP‑aligned**.
