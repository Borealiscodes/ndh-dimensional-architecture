# Authorization of Dormant Projects and Consent Matrix  
### ND‑Safe Collaboration Governance (v1.0)

## 1. Purpose
This document explains how dormant projects inside the UMM ecosystem are authorized, protected, and optionally activated.  
It provides a clear ND‑safe matrix showing how collaboration works without pressure, obligation, or assumption.

This file is **explanatory**, not **activating**.  
It does not request consent.  
It does not trigger growth.  
It does not override sovereignty.

---

## 2. What a Dormant Project Is
A dormant project is a module, organism, or authored artifact that:

- exists in the repo  
- is structurally recognized  
- is protected  
- is sovereign  
- is **not active**  
- is **not expanding**  
- is **not integrated**  
- is **not used in research**  

Dormant means Stillness.

Dormant means Safety.

Dormant means the author remains in full control.

---

## 3. Why Dormant Projects Exist
Dormant projects allow:

- authors to contribute without pressure  
- ideas to rest until the author chooses  
- ND‑safe pacing  
- trauma‑aware boundaries  
- non‑coercive collaboration  
- reversible activation  
- sovereignty over lived experience  

UMM treats dormant projects as **protected seeds**.

Only the author decides if they grow.

---

## 4. Authorization vs Consent (Clear Separation)
**Authorization**  
Explains how things work.  
Provides structure.  
Defines rights.  
Clarifies boundaries.  
Does not activate anything.

**Consent**  
Is a separate action.  
Is voluntary.  
Is revocable.  
Is specific.  
Is sovereign.  
Is never assumed.

This file is **Authorization**, not **Consent**.

---

## 5. ND‑Safe Consent Matrix for Dormant Projects

The matrix below shows how collaboration works in a way that protects ND cognition, trauma boundaries, and author sovereignty.

### **Consent Matrix (ND‑Safe)**

| State | Description | Author Rights | System Behavior |
|------|-------------|---------------|-----------------|
| **Dormant** | Project is resting, unpublished, inactive. | Full sovereignty. No action required. | No activation. No expansion. No research use. |
| **Dormant + Authorized** | Project is structurally recognized and documented. | Author retains full control. | System acknowledges existence but does not activate. |
| **Consent Inquiry** | Author is given optional information. | Author may ignore, decline, or ask questions. | System provides clarity only. No pressure. |
| **Consent Granted (Specific)** | Author chooses a specific activation path. | Author may revoke at any time. | System activates only the chosen path. |
| **Consent Revoked** | Author withdraws consent. | Author sovereignty restored immediately. | System halts activation and returns project to Dormant. |
| **Consent Not Given** | Author does not choose activation. | No obligation. No consequences. | Project remains Dormant indefinitely. |

This matrix ensures:

- no coercion  
- no assumption  
- no pressure  
- no accidental activation  
- no institutional force

## 6. ND‑Safe Collaboration Rules

### **Rule 1 — Author Sovereignty**
The author decides:

- if collaboration happens  
- when collaboration happens  
- how collaboration happens  
- whether collaboration continues  

### **Rule 2 — No Pressure**
Collaboration is always optional.  
Silence = Stillness, not consent.

### **Rule 3 — No Obligation**
Authors are never responsible for:

- maintaining the project  
- expanding the project  
- responding to collaborators  
- producing updates  

### **Rule 4 — Reversible Activation**
Consent can be revoked at any time.  
Revocation is immediate and respected.

### **Rule 5 — ND‑Safe Pacing**
Authors may:

- pause  
- slow down  
- request clarity  
- request simplification  
- request Stillness  

at any time.

### **Rule 6 — Trauma‑Aware Boundaries**
No module may:

- escalate  
- overwhelm  
- force interpretation  
- force activation  
- force collaboration
